# Plan: iOS port

**Date:** 2026-04-16
**Status:** Not started — blocked on Android port
**Goal:** An IPA that runs snowgoons on a tethered iPhone, installed via Xcode with a developer profile. Proof-of-viability, not a shipping product.

## Prerequisites

**The Android port plan must be complete first.** Phases 0–3 of the Android plan produce:
- The VBO/shader renderer (`glpipeline_modern`) running on Linux
- CMake build system with vendored Lua
- HAL lifecycle hooks + asset-accessor interface
- `hal/android/` as a reference for `hal/ios/`

iOS adds Metal/MoltenVK and Xcode plumbing on top of that foundation. ~2 additional engineer-weeks incremental.

## Steps

### Phase 1 — iOS HAL + Xcode project

1. Add `hal/ios/` mirroring `hal/android/` — platform init, input, filesystem (`NSBundle`), timer.
2. Generate Xcode project from CMake (`cmake -G Xcode`). The `.app` bundle is a thin Swift/Obj-C shell wrapping `libwf_game.a`.
3. `UIViewController` entry point with `CAMetalLayer` (native Metal path) or `CAEAGLLayer` (MoltenVK/GLES path). **Decision point — see Open questions.** Start with whichever is lower-friction; the other is a follow-up.
4. Map `UIViewController` lifecycle (`viewWillDisappear`, `viewDidAppear`, etc.) to Phase 2's suspend/resume hooks from the Android plan.
5. Asset pipeline: `[NSBundle mainBundle]` for read-only assets; app-private `NSDocumentDirectory` for any writes.
6. **Verify:** Xcode builds and signs for a connected iPhone (developer profile). App launches without crashing.

### Phase 2 — Graphics backend on iOS

Two options; pick one in Phase 1 and confirm here:

**Option A — GLES via MoltenVK.** Feed Phase 0's `glpipeline_modern` GLES 3.0 calls through MoltenVK → Metal. One code path for Android + iOS; no new shader work.
- If SPIR-V cross-compile is pain-free, ship this.
- If shader translation failures appear, fall back to Option B.

**Option B — Native Metal backend.** Write a `glpipeline_metal` backend implementing the same `SubmitBatch` interface from Phase 0. WF's shader needs are modest (fixed geometry, simple lighting); this is likely smaller-scope than it sounds.

Whichever option is chosen, verify with the golden-image snowgoons diff used in the Android plan's Phase 0.

### Phase 3 — Touch input + full snowgoons run

1. Touch shim identical to Android: on-screen dpad + 4–6 buttons, hit-test each `UITouch`, emit `EJ_BUTTONF_*` bitmask.
2. Optional: MFi gamepad support via `GCController` — same bitmask target as touch; single-day addition.
3. **Verify:** IPA installs via Xcode, snowgoons plays, touch input works. Suspend (home button / swipe), resume, game continues.

### Phase 4 — Second-level smoke test + performance pass

1. Run a non-snowgoons level to confirm level-load path is not snowgoons-hardcoded.
2. Profile with Xcode Instruments (Time Profiler + GPU frame capture): frame time, GPU time, RSS on device.
3. Publish size matrix: IPA size (arm64) alongside the Android APK and Linux binary.
4. Lifecycle torture: force-background 10× in a row; confirm no Metal resource leaks, no crash.

## Open questions

- **MoltenVK or native Metal?** MoltenVK = one code path but extra SPIR-V translation risk. Native Metal = simpler per-platform but doubles backend work. Defer final call until Phase 0's Linux VBO rewrite is running — if WF's rendering is geometry-bound (likely), MoltenVK is fine; if it's shader-complexity-bound, native Metal may be simpler.
- **Minimum iOS version.** Pins Metal availability and Xcode version. Decide when a dev device is in hand. Metal is available since iOS 8; GLES removal from public API was iOS 12. If going MoltenVK, iOS 12+ is the safe floor.
- **MFi gamepad.** `GCController` API maps cleanly to `EJ_BUTTONF_*`; ~1 day of work. Include in Phase 3 as optional.
- **`CAEAGLLayer` removal.** Apple deprecated EAGL and may remove it in a future SDK. If MoltenVK requires EAGL, this is a lurking risk. Track at SDK upgrade time.

## Out of scope

- App Store distribution (own plan when there is a product to ship)
- Android (own plan, must complete first)
- Audio (nothing to port — stubs are stubs)
- Haptics, gyro, camera, mic
- Orientation / responsive layout (landscape-locked letterbox only)
- iPad-specific UI (iPad is just a larger iPhone for v1 purposes; if it runs, great)
- Vulkan backend (MoltenVK counts as Vulkan-via-Metal; a standalone Vulkan backend is a follow-up)
