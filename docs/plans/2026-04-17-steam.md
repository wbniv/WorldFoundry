# Plan: Steam release

**Date:** 2026-04-17
**Status:** Not started
**Goal:** Game available on Steam, launches via the Steam client, Steam overlay works, Steam Input handles controllers. Store page live with a real app ID. Proof-of-distribution, not a polished commercial release.

## Settled decisions

| Decision | Choice |
|----------|--------|
| Platform | Linux first; Windows is its own plan |
| Steamworks SDK | Latest stable at vendor time; prebuilt `libsteam_api.so` shipped alongside binary |
| Steam Input | Yes — replaces raw evdev/xinput gamepad polling; maps to existing `EJ_BUTTONF_*` bitmask |
| Steam DRM | No — optional and adds complexity; skip for v1 |
| Steam Cloud | No — no save games yet |
| Steam Achievements | No — requires design work; own followup |
| Steam Workshop | No |
| Steam Deck | Native Linux binary runs natively on Deck (SteamOS is Linux). No Proton needed. Verify in Phase 3. |
| GL backend | Immediate-mode GL (current Linux renderer) is fine for Steam; the GL rewrite is Phase 0 of the Android plan, not a Steam prerequisite |

## Prerequisites

None. The engine runs on Linux today.

## Steps

### Phase 1 — Steamworks SDK integration
**Estimate: half a day.**

1. Vendor `engine/vendor/steamworks-<ver>/` — SDK headers + `linux64/libsteam_api.so`. Record version + SHA256 in `engine/vendor/README.md`. Do not compile the SDK; it is a prebuilt shared library.
2. Add `WF_ENABLE_STEAM=0|1` flag to `build_game.sh`. When `1`: add `-DWF_ENABLE_STEAM -I"$STEAM_DIR/public"` to CXXFLAGS; link `libsteam_api.so`; compile `hal/linux/steam.cc`.
3. New `wfsource/source/hal/linux/steam.cc`:
   - `SteamAPI_Init()` at game startup (after window open, before game loop). Log result; if Steam client is not running, warn and continue — game must still launch outside Steam for development.
   - `SteamAPI_RunCallbacks()` once per frame in the main loop.
   - `SteamAPI_Shutdown()` at exit.
4. Write `steam_appid.txt` containing the app ID to the working directory; `SteamAPI_Init()` reads it during development.
5. **Verify:** Game launches from Steam client. Steam overlay activates (Shift+Tab). `steam.cc` compiles out cleanly when `WF_ENABLE_STEAM=0`.

### Phase 2 — Steam Input API
**Estimate: 1–2 days.**

Steam Input replaces raw gamepad polling with a platform-agnostic action layer. Controllers appear as action bindings, not raw button indices; Steam handles Xbox/PS/Switch Pro/Steam Controller differences transparently.

1. Define `steam/controller_config.vdf` — In-Game action set with digital actions: `move_left`, `move_right`, `jump`, `attack`, `pause`. Map default bindings for Xbox and PS layouts.
2. New `wfsource/source/hal/linux/input_steam.cc`:
   - `ISteamInput::Init()` at startup; `ISteamInput::Shutdown()` at exit.
   - Each frame: `ISteamInput::RunFrame()`, iterate active controller handles, call `GetDigitalActionData()` per action, translate to `EJ_BUTTONF_*` bitmask.
3. Guard with `#ifdef WF_ENABLE_STEAM`; when Steam is absent the existing raw evdev path stays active.
4. Keyboard and mouse remain raw (Steam Input covers controllers only).
5. **Verify:** Xbox controller and PS4/PS5 controller both produce correct `EJ_BUTTONF_*` input through Steam Input. Remapping in Steam client works.

### Phase 3 — Build, depot, and SteamPipe upload
**Estimate: 1 day.**

1. Create app on Steamworks partner site; note the app ID; fill in `steam_appid.txt`.
2. Configure one depot: Linux x86-64. `arm64` (Steam Deck v2 / future) can be added later without a new depot.
3. Add a `steam/` directory at repo root:
   - `app_build_<appid>.vdf` — SteamPipe build script referencing the Linux depot.
   - `depot_build_<appid>.vdf` — file mapping: binary + `libsteam_api.so` + `cd.iff`.
4. Extend `build_game.sh` with a `dist` target: strip binary, copy `libsteam_api.so`, write `steam_appid.txt`, produce a `dist/` directory ready for SteamPipe.
5. Upload to Steam's `beta` branch via `steamcmd +login ... +run_app_build ...`. Do not set live yet.
6. **Verify (desktop Linux):** Install from Steam beta branch; game launches, overlay works, controllers work.
7. **Verify (Steam Deck):** Sideload or install from beta branch on a Steam Deck; game runs natively (no Proton). Note frame time on Deck's Zen 2 APU.

### Phase 4 — Store page (content, not engineering)
**Estimate: 1–2 days of content work.**

Required before the app can go live. Valve review takes 2–3 business days after submission.

1. Capsule art: 460×215 header, 231×87 small capsule, 616×353 main capsule.
2. Screenshots: minimum 5, 1280×720 or larger.
3. Short description: ≤ 30 words.
4. Long description: HTML-formatted, Steamworks editor.
5. Trailer: optional for an early-access / coming-soon page; required for a full release.
6. Pricing, release date, tags, content survey.
7. Submit for Valve review.

## Out of scope

- Windows port (own plan; starts after Steam Linux ships)
- macOS port (own plan)
- Steam Achievements (requires level-design decisions)
- Steam Cloud saves (no save system yet)
- Steam Workshop / mod support
- Steam Remote Play Together
- Steam DRM
- Steam Deck verified badge (nice-to-have; needs controller+UI audit at 800p)
- Proton compatibility (not needed; native Linux binary runs natively on Deck)
